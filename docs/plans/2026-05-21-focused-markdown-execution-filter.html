Focused Markdown Execution Filter Implementation Plan

Implementation: Use newpowers:executing-plans to implement this plan directly in the current session.

Goal

Change focused markdown exports so Agent Relay and RoboRev reviewer context carries user intent and visible high-level assistant context, while omitting tool execution transcripts and subagent execution detail.

Spec

The full markdown export remains a faithful archive. Only focused markdown, requested with focused=1, changes behavior.

Focused markdown must keep user messages and visible assistant text that survived the existing focused-message selection. Focused markdown must remove execution-only segments before rendering: thinking, tool_call, arguments, tool_body, tool_result, subagent_anchor, and the anchored subagent_session body produced through a tool call.

Focused markdown should not classify tools by name. Write, Edit, Bash, Read, Task, Todo List, MCP tools, and future tool names should all be omitted when they are rendered as tool segments in focused output.

Focused markdown may still include unanchored child sessions that are selected by the existing focused tree filtering, because those are session context rather than the inline body of a tool call. If that becomes too noisy later, handle it as a separate product decision.

Code blocks and skill blocks that appear in visible message text are not execution tags named by this spec. Keep them unless they are part of a tool segment.

Architecture

The markdown renderer already parses messages into segment types in internal/server/export_markdown.go. Replace the current write/edit-specific OmitNoisyToolPayloads option with a focused export option that omits all execution segments during rendering.

The existing focused ordinal filtering in internal/server/export.go should remain responsible for choosing which messages appear. This change should only affect what gets rendered inside the messages selected for focused markdown.

Do not change SQLite persistence, parser typing, or tool-call ingestion. The database stores raw tool-call metadata and JSON; the export layer owns this context-shaping behavior.

Tech Stack

Go server code and tests under internal/server. Verification uses go test, gofmt, git diff --check, and git status/diff inspection before commit.

Task 1: Renderer Unit Spec

Files

  • Modify: internal/server/export_test.go

Happy Path

generateExportMarkdown with default options still renders archival thinking and tool details. With focused execution filtering enabled, it keeps visible text and omits every execution tag and payload.

Unhappy Path

The regression case is any non-write tool, such as Bash or Task, leaking into focused output because the old filter only recognized write-like tools. The test must fail against the current implementation for that reason.

Safeguards

Use payload marker strings that would be damaging in reviewer context if leaked, such as shell commands, command output, task prompts, and child-session content. Assert both absence of tags and absence of payload text.

Steps

  1. Rename TestGenerateExportMarkdown_OmitsNoisyToolPayloads to TestGenerateExportMarkdown_FocusedOmitsExecutionDetails.

  2. Build a mixed assistant message containing visible text, a thinking block, and a non-write tool such as [Bash] with InputJSON, ResultContent, and ResultEvents.

  3. Add an anchored child tree through a Task tool call in a direct generateExportMarkdownTree test, with Depth: "1", so focused filtering must omit subagent_anchor and the anchored subagent body.

  4. Assert default full export includes representative archival tags: <thinking, <tool_call, <arguments, <tool_body, <tool_result, and <subagent_anchor.

  5. Assert focused export keeps visible assistant text such as Here is the high-level result..

  6. Assert focused export excludes <thinking, <tool_call, <arguments, <tool_body, <tool_result, <subagent_anchor, shell command payloads, tool output payloads, task prompts, and anchored child message text.

  7. Run the targeted test before production changes:

    go test ./internal/server -run TestGenerateExportMarkdown_FocusedOmitsExecutionDetails

    Expected: FAIL because the current implementation still renders non-write tool details and thinking segments in focused output.

Task 3: Implement Focused Execution Filtering

Files

  • Modify: internal/server/export_markdown.go

Happy Path

When focused=1, selected messages render without execution-only segments. Full exports without focused=1 are unchanged.

Unhappy Path

If a mixed assistant message contains only thinking and tool segments after filtering, the renderer may emit an empty message wrapper. That is acceptable for this pass if the message survived focused ordinal filtering, but do not introduce malformed XML-like output.

Safeguards

Make the option name describe policy instead of current examples. Remove the per-tool noisy-name classifier so future tools are covered by segment type rather than a maintained allowlist or denylist.

Steps

  1. Rename exportMarkdownOptions.OmitNoisyToolPayloads to OmitExecutionDetails.

  2. In handleMarkdownSession, pass OmitExecutionDetails: focused.

  3. In renderMarkdownMessage, skip markdownSegmentThinking when opts.OmitExecutionDetails is true.

  4. In renderMarkdownMessage, skip markdownSegmentTool when opts.OmitExecutionDetails is true.

  5. Remove the isNoisyMarkdownTool helper and the call to it from renderMarkdownToolSegment.

  6. Run gofmt:

    gofmt -w internal/server/export_markdown.go internal/server/export_test.go internal/server/server_test.go

  7. Run the targeted tests:

    go test ./internal/server -run 'TestGenerateExportMarkdown_FocusedOmitsExecutionDetails|TestMarkdownSessionExport_FocusedOmitsExecutionDetails'

    Expected: PASS.
